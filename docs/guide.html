1. How Winnow thinks

Winnow polls your Fastmail inbox over JMAP and sorts new mail into categories, each mapped to a destination folder. Three principles shape everything it does:

  • Cheap-first. Free header heuristics (List-Unsubscribe, List-Id, Precedence: bulk, known bulk senders) catch the obvious noise. Only genuinely ambiguous mail is sent to Claude, so your API cost stays tiny. The Review tab's Source column shows which path each decision took.
  • Safety bias. When the classifier isn't confident (below your confidence threshold), mail is left in the inbox, unflagged. A stray promo in your inbox is the worst case — a hidden important email never is.
  • Nothing is ever deleted. Every action is a move or a flag, fully reversible from the dashboard.

2. Setup

2.1 Create a Fastmail API token

In Fastmail: Settings → Privacy & Security → Manage API tokens → New API token. Grant these scopes:

  • Mail — read and file your mail
  • Email submission — send the daily digest
  • Sieve / Filters — manage the server-side rules (optional but recommended)

2.2 Generate a dashboard password hash

docker compose run --rm -it winnow hashpw

Enter a password when prompted and copy the printed bcrypt hash (starts with $2a$12$). The -it flag is required — without it the prompt can't read your input.

2.3 Configure winnow.env

Copy .env.example to winnow.env and fill in:

VariableWhat it is
FASTMAIL_TOKENThe API token from step 2.1
ANTHROPIC_API_KEYYour Anthropic API key
APP_PASSWORD_HASHThe bcrypt hash from step 2.2
SESSION_SECRETA random string, e.g. openssl rand -hex 32
Two things that will bite you with the password hash:
  • The file must be named winnow.env, not .env. Docker Compose auto-reads a file named exactly .env and would try to interpolate the $ signs in the hash.
  • Paste the hash raw — no surrounding quotes, no trailing # comment. The compose file loads winnow.env with format: raw so quotes/comments would become literal parts of the value and break login. Just APP_PASSWORD_HASH=$2a$12$...

2.4 Start it

docker compose up -d

The first run seeds the preset categories and starts in dry-run mode — it previews classifications without moving anything. That's exactly what you want for the first week.

3. Reaching the dashboard

The app binds to 0.0.0.0:8080 inside the container, published on a host port (e.g. 8083). Two private ways in:

  • Tailscale (primary). Reach it at your server's Tailnet address, e.g. http://your-server:8083. Note the host port must bind to all interfaces (not 127.0.0.1) for Tailscale traffic to reach it.
  • Cloudflare Tunnel (optional public path). Start with docker compose --profile tunnel up -d and gate it with Cloudflare Access. Winnow additionally verifies the Access JWT on every request. See the README for the full setup.

5. Dry-run: what to check before going live

Dry-run governs only the automatic triage pipeline — the scheduled classification of new mail (and the sweep preview). While it's on, that pipeline logs what it would do without moving, flagging, or marking anything read.

Dry-run does NOT cover explicit actions you take yourself. These act for real even while the dry-run badge is showing — by design, since you asked for them:
  • Sweep inbox (apply) — moves the existing backlog into folders.
  • Move & teach (Review tab) — re-files that one email over JMAP.
  • Approving & applying Sieve rules (Rules tab) — writes the rule to Fastmail. From then on Fastmail files matching mail server-side at delivery, completely outside Winnow and its dry-run setting. To pause that, revert or remove the managed rule.
  • Unsubscribe (Unsubscribe / Senders tabs) — a One-Click POST or mailto: is actually sent to the sender.
  • Teach (Review tab) — records sender stats and allow/deny overrides. It doesn't move mail, but the learning persists and shapes future triage.
  • Daily digest — still emailed; it's controlled by the separate "daily digest" toggle, not by dry-run.
In short: dry-run is a safety net for Winnow's autonomous filing, not a global "do nothing" switch. Anything you click is a deliberate action and runs.

Before switching dry-run off (handing the automatic pipeline the keys), satisfy these:

  • No false negatives. Scan for any genuinely important email assigned to a bulk category (Promotional/Newsletters). This is the one that matters. If you find one, add that sender to the allow list in the Senders tab.
  • Heuristics are catching the obvious bulk. In the Source column, promos and newsletters with unsubscribe headers should read heuristic (free). If clear bulk shows Claude, you're paying unnecessarily.
  • Confidence threshold feels right. Too many legit emails marked low and kept back? Lower the threshold. Junk slipping through as low-confidence? Raise it.
  • Folder mappings make sense for your inbox (Categories tab).

How long?

Gate on confidence, not the calendar. Don't wait passively — run Sweep inbox (preview) to classify your entire existing inbox at once and get a large sample to judge immediately. Then keep dry-run triage running for about one to two weeks so your periodic senders (weekly newsletters, monthly statements, billing) each show up at least once and you can allow-list the important ones before they'd ever move.

Preview is a true dry read. Running Sweep inbox (preview) never moves mail, never marks it processed, and won't inflate the rule-learning counters or pile up duplicate log entries — so you can re-run it as often as you like. The only cost is that each run re-classifies the ambiguous mail, which re-bills those Claude calls (heuristic-caught mail stays free).

6. Going live (and the existing-inbox backlog)

Turning off dry-run does not retroactively apply past decisions. Those decisions are point-in-time previews, not a queue. Normal triage advances its position each cycle, so once dry-run is off it acts only on new mail arriving from that point forward. The mail already sitting in your inbox won't move on its own.

To file mail that's already in your inbox, use Sweep inbox (apply) — the dedicated backlog path that processes the existing inbox in checkpointed chunks of 75, skipping anything already handled. Both sweep buttons look at the Inbox only (mail already in other folders is left alone) and process up to 10,000 messages per run, newest first. If your inbox is smaller than that, the sweep covers all of it — old mail included. (If it's larger and you need the oldest beyond 10,000, ask and the cap can be raised or made a setting.)

Sweep (apply) overrides the dry-run toggle. It moves mail immediately even if the dry-run checkbox is still on. Treat that button as live.

Recommended go-live sequence

  1. Run Sweep inbox (preview) and review the full-inbox proposals.
  2. Let triage run in dry-run ~1 week to catch periodic senders; allow-list anything important.
  3. When happy: uncheck dry-run in Settings (live triage for new mail) and run Sweep inbox (apply) once to clear the existing backlog into folders.

Because nothing is ever deleted, the cost of switching a little early is just the occasional misfiled email you correct with Move & teach.

8. Sieve rules — server-side filtering

As Winnow sees a sender domain land in the same category repeatedly, it proposes a Fastmail Sieve rule in the Rules tab. Approve it and that domain is then filed server-side at delivery — free, instant, and working even when Winnow is offline.

  • Winnow only ever edits its own delimited managed block. Your hand-made Fastmail rules are left byte-for-byte untouched.
  • Rules are consolidated by domain (one rule per category holding a domain list), so the managed rule count stays at a handful no matter how many domains accumulate.
  • Every change is validated before activation, and the prior script is backed up with one-click revert.

9. Assisted unsubscribe

The Unsubscribe tab ranks bulk senders that expose a List-Unsubscribe header. For each you choose:

  • Unsubscribe — for RFC 8058 One-Click and mailto: methods, Winnow executes it after approval. Bare HTTPS links are shown for you to open manually and are never auto-visited (that's a tracking/phishing vector).
  • Keep receiving — file it to a category instead. Reversible: you can unsubscribe a kept sender later, even months on, because the unsubscribe method is persisted when first seen.

After you unsubscribe, Winnow watches for continued mail. If a sender keeps emailing past the verify window, it flags them and offers a Sieve-rule fallback.

10. The daily digest

Once a day Winnow emails you a summary of what it filed and what needs your attention, sent from your own address to yourself (derived from the JMAP session — never hardcoded). It doubles as a heartbeat: if the digest stops arriving, something is wrong. Any persistent errors (expired token, repeated API failures, spend cap hit) appear as a line in the digest and as a banner in the dashboard.

11. Maintenance & backups

  • Backups. All learned state — categories, sender stats, rule backups, decision log, unsubscribe metadata — lives in the SQLite database on the winnow-data volume. Back it up with a volume copy or sqlite3 winnow.db ".backup backup.db".
  • Decision log auto-prunes daily per your retention setting.
  • Upgrades. Schema migrations run automatically on startup, so pulling a new image and restarting is safe.

12. Troubleshooting

SymptomCause & fix
The "..." variable is not set warning on startup Compose is interpolating a $ in your env file. Ensure the file is named winnow.env and the compose env_file uses format: raw.
Login always fails The stored hash doesn't match your password, often because it was pasted with quotes or a trailing comment. Regenerate with hashpw and paste it raw, with no quotes or comment.
jmap method error "invalidArguments" during triage A JMAP request was malformed for Fastmail. Winnow uses header:*:asRaw for header properties; make sure you're on a current build.
Dashboard unreachable over Tailscale The host port is bound to 127.0.0.1. Bind to all interfaces (or your Tailscale IP) so Tailnet traffic can reach it.
Mail in the inbox isn't being filed after going live Normal triage only acts on new mail. Run Sweep inbox (apply) for the existing backlog (see section 6).
